问题 1：R 代码集成方式

AnalysisAgent 的工具内部如何执行生物医学分析（差异表达、生存分析、免疫关联、药物筛选）？

三种路线对比

A

Python subprocess → Rscript temp.R

每次分析写临时 .R 文件 → subprocess 调 Rscript → 解析 stdout/RDS 输出

Pros

  • 复用 ITIP/CSTB 已有的 R 代码（Phase C/E 脚本）
  • 结果和已验证的 ITIP 计算完全一致
  • 不需要额外依赖（R 4.5.2 已配置）

Cons

  • 每次运行 30-120s，面试 demo 体验差
  • Windows Rscript 偶发 segfault (Rule-R-001)
  • 错误处理复杂：R 报错是纯文本，Python 难以结构化解析
  • 需要确保临时文件不冲突（并发时有风险）
B

rpy2 桥接

Python 直接调 R 函数：rpy2.robjects.r('source("cox.R")')

Pros

  • 无临时文件，无 subprocess 开销
  • Python/R 之间直接传 DataFrame
  • 错误可以抛 Python 异常

Cons

  • rpy2 在 Windows 上安装极困难（需要编译 R_HOME）
  • 内存占用大：嵌入式 R 进程常驻 500MB+
  • 你现有的 ITIP/CSTB R 代码依赖特定 library 路径（D:/R/library），rpy2 不一定能正确初始化
  • 违反 RULE-R-003：引入新依赖
C

预计算缓存 + 实时 Python 查询 03- 推荐

ITIP/CSTB 的分析结果已缓存为 JSON → Agent "工具调用" = 查询缓存 + Python 实时计算 + LLM 解释

Pros

  • Demo 秒级响应，不依赖 R 运行时
  • 缓存数据已经过验证（ITIP Phase C/E 的结果）
  • Agent 的智能体现在"选什么方法、怎么解释"，而非"等 R 跑完"
  • 符合 Rule-R-001：完全避免 Rscript segfault

Cons

  • 仅限于有缓存的分析 — 新的基因需要预计算
  • 面试时需诚实说"核心分析是预计算的"
  • 缓存数据需要一次性预计算脚本生成
路线 C 的 Agent 工具调用流程
Think: "要验证 CSTB 在 CRC 中是否差异表达"
▼
Act: tool_call("run_differential_expression", {
  "gene": "CSTB",
  "dataset": "TCGA-COAD"
})
▼
Observe: 工具内部：读缓存 JSON → 返回 "CSTB logFC=2.3, adj.P=1.2e-15"

我的推荐：C 为主 + A 为补充

核心分析（差异表达、生存分析、免疫关联、药物筛选）用 预计算缓存（路线 C）。但保留一个 轻量 subprocess 通道（路线 A）处理缓存未覆盖的情况：当 Agent 需要分析一个不在缓存中的新基因时，降级为简单的 Python pandas/scipy 计算（t-test、Spearman correlation），不调 R。
