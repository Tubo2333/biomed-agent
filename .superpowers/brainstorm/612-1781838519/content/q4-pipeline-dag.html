问题 4：Pipeline 固定顺序 vs 可扩展 DAG

4 Agent 串行够不够？OrchestrationAgent 内部的 DAG 能多灵活？

两个层级的设计决定

外层：Pipeline 编排器 — 固定 4 Agent 串行
Phase 1: LiteratureAgent.run(question)
▼ LiteratureReview
Phase 2: OrchestrationAgent.plan(review)
▼ AnalysisPlan (DAG)
Phase 3: AnalysisAgent.execute(plan)
▼ AnalysisResults
Phase 4: ReportAgent.generate(...)
▼ Report

✅ 科研工作流天然是顺序的（先看文献→再设计→再实验→再写报告）
✅ 简单、可理解、容易调试、容易测试
✅ 适合面试 demo：每一步的中间输出都可展示

内层：OrchestrationAgent.plan() — 动态 DAG

输入：LiteratureReview（含 "CSTB通过M2巨噬细胞介导免疫抑制" 假设）

LLM 推理 → 输出 DAG：

Node 1: diff_expression(CSTB)
  └→ Node 2: immune_correlation(CSTB, M2_markers)
Node 3: cox_survival(CSTB, M2_high_vs_low)
  └→ depends on [Node 1, Node 2]
Node 4: drug_screening(CSTB)
  └→ parallel: independent of others

✅ 不同 LiteratureReview → 不同 DAG（不是硬编码）
✅ 体现 JD "LLM 规划"+"多步决策"
✅ DAG 内部支持并行（max 2 并行，受限于 Python 单进程）

关键对比

维度 外层 Pipeline 内层 DAG
结构 固定 4 Agent 串行 LLM 动态生成，可变节点数/边
变化来源 无变化（代码即结构） 随 LiteratureReview 内容变化
执行顺序 拓扑序（依赖 S1→S2→S3→S4） 拓扑序 + 并行（无依赖节点并行）
失败处理 单 Agent 失败 → 整体失败 单节点失败 → F2/F3/F4 恢复
过度工程风险 低 需要克制 — 不需要完整的 DAG 执行引擎

推荐：外层固定串行 + 内层 LLM 动态 DAG

与你现有的 03- 设计文档完全一致，也是 Spatial master.py 的升级方向：从硬编码拓扑 → LLM 推理拓扑。
